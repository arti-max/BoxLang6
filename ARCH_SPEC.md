# BoxLang Architecture Specification
## Битово-ориентированный ISA-дескриптор

**Версия:** 1.1  
**Назначение:** Описание формата arch JSON файлов для компилятора BoxLang.  
Этот документ одновременно является технической документацией и
спецификацией для любого инструмента (включая AI), работающего с
архитектурными конфигами BoxLang.

---

## Концепция

Традиционные компиляторы жёстко зашивают логику генерации кода
под конкретную архитектуру в Python/C++ коде. BoxLang использует
другой подход:

> **Компилятор — это интерпретатор битовых полей.**
> Он не знает ничего про x86, ARM, или любую другую архитектуру.
> Вся архитектурная логика описана в JSON файле.

Это означает:
- Добавить новую архитектуру = написать JSON файл, не трогая компилятор
- Архитектуры могут кардинально отличаться (4-бит без стека vs 16-бит x86)
- JSON файл является единственным источником истины об архитектуре

---

## Файловая структура

```
arch/
├── x16.json       # x86-16 (x16-PRos)
├── a4004.json     # кастомный 4-битный процессор
├── schema.json    # JSON Schema для валидации конфигов
└── ARCH_SPEC.md   # этот файл
```

## Верхний уровень JSON

```json
{
  "name":    "x16",       // уникальное имя архитектуры
  "bits":    16,          // разрядность: 2, 4, 8, 16, 32
  "endian":  "little",    // "little" или "big"

  "registers":           { ... },   // таблица регистров
  "calling_convention":  { ... },   // соглашение о вызовах
  "var_storage":         "stack",   // стратегия хранения переменных
  "ops":                 { ... },   // абстрактные операции
  "instructions":        { ... },   // таблица инструкций с битовым описанием
  "syscalls":            { ... },   // системные вызовы
  "prologue":            [ ... ],   // пролог функции
  "epilogue":            [ ... ],   // эпилог функции
  "output":              { ... }    // настройки выходного файла
}
```

## 1. Регистры (`registers`)

Каждый регистр — объект только с двумя полями: `id` и `bits`.
Никаких специальных полей (`half`, `segment`, и т.п.) не существует —
вся семантика регистра выражается через его `id` в битовых полях инструкций.

```json
"registers": {
  "ax": { "id": 0, "bits": 16 },
  "bx": { "id": 3, "bits": 16 },
  "al": { "id": 0, "bits": 8  },
  "ah": { "id": 4, "bits": 8  },
  "a":  { "id": 0, "bits": 4  }
}
```

| Поле   | Тип     | Обязательно | Описание |
|--------|---------|-------------|----------|
| `id`   | integer | да          | Число, используемое при энкодинге инструкции |
| `bits` | integer | да          | Разрядность: 4, 8, 16, 32 |

### Почему `ah` имеет `id: 4`, а не `id: 0`?

В x86 опкод `MOV reg8, imm8` вычисляется как `0xB0 + reg_id`.
- `MOV al, imm8` → `0xB0 + 0` = `0xB0`
- `MOV ah, imm8` → `0xB0 + 4` = `0xB4`

Значит `ah.id = 4` — это не смысловая "позиция" внутри ax,
а просто число которое правильно упаковывается в битовое поле
конкретной инструкции. Компилятор не знает про "половинки" —
он просто подставляет `reg.id` в нужные биты.

### Регистры в других архитектурах

Если в архитектуре есть аналогичное деление регистров на части —
просто дай им правильные `id`, которые корректно упакуются
в битовые поля твоих инструкций. Никаких дополнительных полей
в спецификации регистра не предусмотрено и не нужно.

## 2. Соглашение о вызовах (`calling_convention`)

```json
"calling_convention": {
  "args":        ["ax", "bx", "cx", "dx"],
  "return":      "ax",
  "stack_align": 2,
  "preserved":   ["bp", "si", "di"]
}
```

| Поле          | Описание |
|---------------|----------|
| `args`        | Регистры для передачи аргументов (по порядку) |
| `return`      | Регистр для возвращаемого значения ("рабочий регистр") |
| `stack_align` | Выравнивание стека в байтах (0 если стека нет) |
| `preserved`   | Регистры, которые функция обязана сохранить |

Для архитектур без стека (bare metal, 4-бит):
```json
"calling_convention": {
  "args":        ["a"],
  "return":      "a",
  "stack_align": 0,
  "preserved":   []
}
```

## 3. Хранение переменных (`var_storage`)

Определяет как компилятор размещает локальные переменные.

### Вариант A: стек (`"stack"`)

Переменные живут на стеке. Компилятор генерирует выделение места
и обращается к переменным через смещение от BP.

```json
"var_storage": "stack",
"var_alloc": {
  "insn": "sub", "variant": "reg16_imm",
  "args": ["sp", "$size"]
},
"var_load": {
  "insn": "mov", "variant": "reg_bp_off",
  "args": ["$reg", "$bp_offset"]
},
"var_store": {
  "insn": "mov", "variant": "bp_off_reg",
  "args": ["$bp_offset", "$reg"]
}
```

Биндинги:
- `$size`      — размер переменной в байтах
- `$bp_offset` — смещение от BP (отрицательное: -2, -4, ...)
- `$reg`       — рабочий регистр из `calling_convention.return`

### Вариант B: регистры (`"registers"`)

Переменные живут в регистрах. Компилятор распределяет их
по списку `var_regs`. Если переменных больше чем регистров —
ошибка компиляции.

```json
"var_storage": "registers",
"var_regs": ["b", "c", "d"]
```

## 4. Абстрактные операции (`ops`)

Это **контракт** между компилятором и архитектурой.
Компилятор вызывает операции по имени, не зная как они реализованы.
Архитектура описывает реализацию через последовательность инструкций.

```json
"ops": {
  "load_imm": {
    "desc": "Загрузить константу в рабочий регистр",
    "steps": [
      { "insn": "mov", "variant": "reg16_imm16", "args": ["ax", "$value"] }
    ]
  }
}
```

Каждая операция — список шагов `steps`. Один шаг:
```json
{
  "insn":    "имя инструкции из таблицы instructions",
  "variant": "имя варианта",
  "args":    ["аргументы; $биндинги заменяются значениями при вызове"]
}
```

### Биндинги в args

Аргументы шага могут быть:
- Именем регистра: `"ax"`, `"a"`, `"bp"`
- Числовой константой: `"0"`, `"2"`
- Биндингом: `"$value"`, `"$label"`, `"$code"` — подставляется компилятором
- Inline-константой: `"const:0x4C"` — жёстко зашитое число прямо в шаге

### Полная таблица операций

| Операция             | Биндинги                  | Обязательна | Описание |
|----------------------|---------------------------|-------------|----------|
| `load_imm`           | `$value`                  | да          | Загрузить константу в рабочий регистр |
| `load_var`           | `$bp_offset` / `$address` | да (stack)  | Загрузить переменную |
| `store_var`          | `$bp_offset` / `$address` | да (stack)  | Сохранить переменную |
| `copy_reg`           | `$src`, `$dst`            | да          | Скопировать регистр в регистр |
| `push_reg`           | `$reg`                    | да (stack)  | Сохранить регистр на стеке |
| `pop_reg`            | `$reg`                    | да (stack)  | Восстановить регистр со стека |
| `add`                | `$right`                  | да          | `work_reg += $right` |
| `sub`                | `$right`                  | да          | `work_reg -= $right` |
| `mul`                | `$right`                  | нет         | Умножение (опционально) |
| `div`                | `$right`                  | нет         | Деление (опционально) |
| `neg`                | —                         | нет         | Унарный минус |
| `cmp`                | `$right`                  | да          | Сравнение (устанавливает флаги) |
| `call`               | `$label`                  | да          | Вызов функции |
| `ret`                | —                         | да          | Возврат из функции |
| `jump`               | `$label`                  | да          | Безусловный прыжок |
| `jump_if_zero`       | `$label`                  | да          | Прыжок если рабочий регистр == 0 |
| `jump_if_not_zero`   | `$label`                  | да          | Прыжок если != 0 |
| `jump_if_less`       | `$label`                  | да          | Прыжок если < (после cmp) |
| `jump_if_greater`    | `$label`                  | да          | Прыжок если > |
| `jump_if_less_eq`    | `$label`                  | да          | Прыжок если <= |
| `jump_if_greater_eq` | `$label`                  | да          | Прыжок если >= |
| `syscall_exit`       | `$code`                   | да          | Завершение программы |

Операции `mul`, `div`, `neg` — опциональные. Если отсутствуют в JSON —
компилятор выдаёт ошибку при попытке использовать `*`, `/` или унарный `-`.

## 5. Инструкции (`instructions`)

Каждая инструкция имеет один или несколько вариантов.
Каждый вариант описывает **битовую структуру** результирующих байт.

```json
"instructions": {
  "имя_инструкции": {
    "desc": "описание (опционально)",
    "variants": {
      "имя_варианта": {
        "size_bits": 24,
        "fields": [
          { "bits": "23:16", "source": "const:0xB8" },
          { "bits": "15:0",  "source": "arg:1:imm"  }
        ]
      }
    }
  }
}
```

### 5.1 `size_bits`

Полный размер инструкции в битах. Кратность 8 обязательна,
исключение — 4-битные архитектуры где допустимо значение `4`.

```
4  → 1 нибл     (4-бит архитектуры)
8  → 1 байт
16 → 2 байта
24 → 3 байта
32 → 4 байта
```

### 5.2 `bits` — диапазон битов поля

Формат: `"hi:lo"`. Нумерация с 0 от **младшего** бита.

```
Пример для size_bits=24:

  бит: 23  22  21  20  19  18  17  16  | 15 .. 8 | 7 .. 0
              старший байт               средний   младший

"23:16" → весь старший байт
"18:16" → 3 бита (18, 17, 16)
"15:0"  → два младших байта
"7:4"   → старший нибл младшего байта
"3:0"   → младший нибл младшего байта
"0:0"   → один бит
```

### 5.3 `source` — источник значения поля

#### `const:VALUE`

Константа, вшитая прямо в JSON.
`VALUE` — hex (`0xB8`) или dec (`184`).

```json
{ "bits": "7:4", "source": "const:0x1" }
```

Применение: опкоды, фиксированные биты (mod=11 в ModRM x86),
magic bytes, hardcoded поля.

**Важно:** ModRM в x86 — это просто три битовых поля в одном байте.
Никакого специального типа `modrm` не существует — он описывается
обычными `const` и `arg` полями:

```
ModRM = 

mod=11 (reg-to-reg), reg_field=0 (для ADD /0), rm=номер регистра:

{ "bits": "23:22", "source": "const:0x3"    }  // mod = 11
{ "bits": "21:19", "source": "const:0x0"    }  // reg_field = 0
{ "bits": "18:16", "source": "arg:0:reg.id" }  // rm = номер регистра
```

---

#### `arg:N:TYPE`

Значение из N-го аргумента инструкции (нумерация с 0).

| TYPE       | Формула                        | Описание |
|------------|--------------------------------|----------|
| `imm`      | `value`                        | Число как есть |
| `imm.lo8`  | `value & 0xFF`                 | Младший байт |
| `imm.hi8`  | `(value >> 8) & 0xFF`          | Старший байт |
| `imm.lo4`  | `value & 0xF`                  | Младший нибл |
| `imm.hi4`  | `(value >> 4) & 0xF`           | Старший нибл |
| `reg.id`   | `registers[name].id`           | Числовой id регистра |
| `reg.bits` | `registers[name].bits`         | Разрядность регистра |

```json
{ "bits": "18:16", "source": "arg:0:reg.id"  }
{ "bits": "15:0",  "source": "arg:1:imm"     }
{ "bits": "3:0",   "source": "arg:0:imm.lo4" }
```

---

#### `label:TYPE`

Ссылка на метку. На первом проходе записывается как `0`,
добавляется в таблицу релокаций. Патчится на втором проходе.

| TYPE    | Описание |
|---------|----------|
| `rel8`  | Знаковое 8-бит смещение от конца инструкции |
| `rel16` | Знаковое 16-бит смещение от конца инструкции |
| `abs8`  | Абсолютный 8-бит адрес |
| `abs16` | Абсолютный 16-бит адрес |
| `abs4`  | Абсолютный 4-бит адрес (для 4-бит архитектур) |

```json
{ "bits": "15:0", "source": "label:rel16" }
{ "bits": "3:0",  "source": "label:abs4"  }
```

### 5.4 Примеры инструкций

#### x86-16: `MOV reg16, imm16` (опкод `0xB8 + reg.id`)

```
Байт 1: 1011 1    →  биты 23..19 = 10111 = 0x17? нет...

0xB8 = 1011 1000
Старшие 5 бит: 10111 = 23 (0x17) — нет, считаем точно:
0xB8 >> 3 = 0x17 = 23 = 10111b  ✓
Младшие 3 бита: номер регистра
```

```json
"mov": {
  "variants": {
    "reg16_imm16": {
      "size_bits": 24,
      "fields": [
        { "bits": "23:19", "source": "const:0x17"   },
        { "bits": "18:16", "source": "arg:0:reg.id" },
        { "bits": "15:0",  "source": "arg:1:imm"    }
      ]
    }
  }
}
```
Для `mov ax, 42`: `B8 2A 00` ✓
Для `mov bx, 42`: `BB 2A 00` ✓  (bx.id=3, 10111_011 = 0xBB)

---

#### x86-16: `ADD reg16, imm16` (опкод `0x81`, ModRM `/0`)

```
Байт 1: 0x81
Байт 2: ModRM = [reg]
  mod=11 → const 0x3
  reg_field=0 (ADD) → const 0x0
  rm = номер регистра
Байт 3-4: imm16
```

```json
"add": {
  "variants": {
    "reg16_imm16": {
      "size_bits": 32,
      "fields": [
        { "bits": "31:24", "source": "const:0x81"   },
        { "bits": "23:22", "source": "const:0x3"    },
        { "bits": "21:19", "source": "const:0x0"    },
        { "bits": "18:16", "source": "arg:0:reg.id" },
        { "bits": "15:0",  "source": "arg:1:imm"    }
      ]
    }
  }
}
```
Для `add ax, 5`: `81 C0 05 00` ✓  (C0 = 11_000_000)
Для `add bx, 5`: `81 C3 05 00` ✓  (C3 = 11_000_011)

---

#### x86-16: `SUB reg16, imm16` (опкод `0x81`, ModRM `/5`)

Отличается только `reg_field = 5` (SUB):
```json
"sub": {
  "variants": {
    "reg16_imm16": {
      "size_bits": 32,
      "fields": [
        { "bits": "31:24", "source": "const:0x81"   },
        { "bits": "23:22", "source": "const:0x3"    },
        { "bits": "21:19", "source": "const:0x5"    },
        { "bits": "18:16", "source": "arg:0:reg.id" },
        { "bits": "15:0",  "source": "arg:1:imm"    }
      ]
    }
  }
}
```
Для `sub ax, 5`: `81 E8 05 00` ✓  (E8 = 11_101_000)

---

#### x86-16: `MOV reg8, imm8` (ax-половинки)

`al.id=0`, `ah.id=4` — `id` уже содержит правильное число для опкода.
Формула опкода: `0xB0 + reg.id`

```json
"mov": {
  "variants": {
    "reg8_imm8": {
      "size_bits": 16,
      "fields": [
        { "bits": "15:11", "source": "const:0x16"   },
        { "bits": "10:8",  "source": "arg:0:reg.id" },
        { "bits": "7:0",   "source": "arg:1:imm"    }
      ]
    }
  }
}
```
Для `mov al, 0x4C`: `B0 4C` ✓  (0x16 << 3 | 0 = 0xB0)
Для `mov ah, 0x4C`: `B4 4C` ✓  (0x16 << 3 | 4 = 0xB4)

---

#### 4-бит: `LDA imm4`

```json
"lda": {
  "variants": {
    "imm4": {
      "size_bits": 8,
      "fields": [
        { "bits": "7:4", "source": "const:0x1"      },

    }
  }
}
```
Для `lda 5`: `0x15` = `0001 0101` ✓

#### 4-бит: `JMP addr4`

## 6. Системные вызовы (`syscalls`)

Последовательность шагов для системных операций.
Биндинги передаются компилятором при вызове.

```json
"syscalls": {
  "exit": {
    "desc": "Завершение программы",
    "steps": [
      { "insn": "mov", "variant": "reg8_imm8", "args": ["ah", "const:0x4C"] },
      { "insn": "mov", "variant": "reg8_imm8", "args": ["al", "$code"]       },
      { "insn": "int", "variant": "imm8",      "args": ["const:0x21"]        }
    ]
  }
}
```

Для bare metal без ОС:
```json
"syscalls": {
  "exit": {
    "steps": [
      { "insn": "hlt", "variant": "plain", "args": [] }
    ]
  }
}
```

`"const:VALUE"` в `args` — inline-константа, не биндинг.

```json
"prologue": [],
```

## 9. Алгоритм компилятора

### Проход 1: emit

```
для каждого узла AST:
    определить абстрактную операцию (load_imm, call, ...)
    найти ops[операция] в arch JSON
    для каждого шага ops[операция].steps:
        найти instructions[insn].variants[variant]
        result = 0  (целое число, size_bits бит)
        для каждого field в variant.fields:
            [hi, lo] = parse("hi:lo")
            width = hi - lo + 1
            value = resolve_source(field.source, args)
            если source = label:* →
                добавить Relocation(buf_offset, hi, lo, label, type, insn_end)
                value = 0
            result |= (value & ((1 << width) - 1)) << lo
        записать result как (size_bits // 8) байт в буфер
```

### Проход 2: resolve relocations

```
для каждого Relocation:
    target = labels[relocation.label]
    если rel8 или rel16:
        delta = target - relocation.insn_end
        проверить диапазон: rel8 → [-128..127], rel16 → [-32768..32767]
    если abs:
        delta = target + origin
    width = relocation.bit_hi - relocation.bit_lo + 1
    mask  = (1 << width) - 1
    byte_offset = relocation.buf_offset + (relocation.bit_lo // 8)
    упаковать (delta & mask) в биты [bit_hi:bit_lo] буфера
```

### Структура Relocation

```python
@dataclass
class Relocation:
    buf_offset: int   # байтовое смещение начала инструкции в буфере
    bit_hi:     int   # старший бит поля внутри инструкции
    bit_lo:     int   # младший бит поля
    label:      str   # имя метки
    label_type: str   # "rel8", "rel16", "abs8", "abs16", "abs4"
    insn_end:   int   # buf_offset + size_bits//8 (для rel вычислений)
```
